## STEP 9 — Create a deploy Skill with semantic versioning

Create a deployment skill script at /scripts/deploy.sh that:

### Version management
- Read the current version from a VERSION file in the project root
  (create it with initial value 1.0.0 if it does not exist)
- Accept an optional argument: patch, minor, or major
  - patch = 1.0.0 → 1.0.1 (bug fixes, content updates)
  - minor = 1.0.0 → 1.1.0 (new features)
  - major = 1.0.0 → 2.0.0 (breaking changes)
  - default (no argument) = bump patch automatically

### Service worker cache busting
- Before deploying, automatically update the CACHE version 
  in /bestanden/pwa/sw.js from:
    const CACHE = 'vod-v1';
  to the new version, for example:
    const CACHE = 'vod-v1.0.1';
  This forces all users to get fresh content after an update.

### manifest.json version
- Also update the version field in manifest.json if present

### Git tagging
- Stage all changed files: git add .
- Commit with message: "Release v[new-version]"
- Create a git tag: git tag v[new-version]
- Push branch and tags: git push && git push --tags

### Firebase deploy
- Run firebase deploy --message "v[new-version]"

### Rollback capability
- Add a rollback function: deploy.sh rollback
  that runs: firebase hosting:rollback
  and reverts the VERSION file and git tag

### Changelog
- Auto-append an entry to CHANGELOG.md with:
  - Version number
  - Date and time
  - Deploy type (patch/minor/major)
  - Last git commit message as description

### Usage instructions
The script should be runnable as:
  ./scripts/deploy.sh          → auto patch bump + deploy
  ./scripts/deploy.sh minor    → minor version bump + deploy
  ./scripts/deploy.sh major    → major version bump + deploy
  ./scripts/deploy.sh rollback → revert to previous Firebase version

Make the script executable with chmod +x scripts/deploy.sh
